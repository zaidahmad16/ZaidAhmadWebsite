The problem

My friend in Engineering showed me the course tree his department gives to first-years. A PDF, just a grid. Every required course, what unlocks what, four years on one page. His whole degree, readable at a glance.

My program didn't have one. So you just pieced it together yourself. Academic calendar in one tab, prerequisite lists buried in paragraph form in another. I'd done it. It takes longer than it should, you still miss things, and it shouldn't work that way.

I built a map for every Carleton program. I had no idea how many things were going to break first.

Students helped: 700+
Programs indexed: 304+
Courses indexed: 3,218+

What it does

Pick a program, get a four-year prerequisite map. Courses in a grid by year and semester. Fall left, Winter right. Arrows for prerequisites, dashed for concurrent. Color tells you the course type: black border for required, green for Math and Science, orange for electives, red for complementary studies. The scheme matches Carleton Engineering's own PDFs so students already know what it means.

Click a course and you get the description, credit weight, which semesters it runs, and professor info with RateMyProfessors ratings updated monthly. Elective slots show up as placeholders so the grid doesn't look wrong where a semester has open choices. It works on mobile. More students check it on their phones than I expected, mid-registration, trying to figure out whether they actually qualify for something.

How it got here

Scraping Carleton Central

All the data comes from Carleton Central. Getting it clean took way longer than expected and involved throwing away the first scraper entirely.

I started with Beautiful Soup. It's well documented and gets you somewhere fast, which is exactly what it did, until the scraper got complex enough that "somewhere fast" wasn't where I needed to be. Selector syntax got clunky. Performance dragged. When you're parsing prerequisites formatted differently per department and per year the course was added, you need more precision than Beautiful Soup gives you cleanly.

I switched to Parsel, which is what Scrapy runs on. XPath let me navigate inconsistent HTML without stacking workarounds on top of each other. Rewriting working code hurts. Still the right call.

The prerequisite parsing was the worst part. Prerequisites at Carleton come in every shape: AND chains, OR choices, nested combinations, minimum grades, year standing thresholds, all in plain English prose with no consistent format. "Prerequisite: COMP 1405 or COMP 1406" from one department. "Prerequisite(s): One of COMP 1405, COMP 1406 with a grade of C- or higher" from another. Some list year standing instead of specific courses. Some formats showed up exactly once in the entire catalog.

Every time I thought I'd handled every case, a new program found another one I hadn't. I lost count of how many times I rewrote pieces of the parser. For semester offerings, a lot of courses have no explicit data in the system. Rather than leaving them off the map, I use a heuristic. Odd course numbers tend to run in Fall, even in Winter. It holds up well enough across most departments. Real data takes priority where it exists.

Database & backend

Prerequisite logic doesn't map neatly to rows and columns. Courses can have multiple prerequisites that AND or OR together, and it nests. I used JSONB in PostgreSQL to store the AND/OR tree directly rather than building a separate prerequisites table with its own relationship logic. Database is on Neon, backend is FastAPI.

Setting them up had its moments. I once pasted a database connection string into .env twice, merging two URLs into one broken string. The error looked like an application crash. I stared at code that was completely fine for a while before thinking to check the environment variable itself. Another time I accidentally pasted my actual Neon password into a chat window while asking for help with something unrelated. Stopped everything and rotated credentials. The kind of afternoon where you just close the laptop for a bit.

Frontend

Frontend is Next.js. The map is React Flow. Node rendering, edges, zoom, pan. For positioning courses I used dagre, which calculates layout from the graph automatically. Without it I'd have been hardcoding positions for hundreds of nodes.

Tailwind v4 broke things early. Deep incompatibilities with Turbopack. Classes not applying, builds failing, nothing pointing clearly at Tailwind. I spent too long assuming the issue was in my Next.js config. Downgraded to v3, fixed. Still pinned there.

Then the dev server started dying randomly. Turned out I had two next.config.js files in different locations. Next.js loaded both, Turbopack spun up duplicate processes, and memory went sideways. Deleted the extra file, capped the Node.js heap at 2GB. Stable after that.

The arrow problem

This took longer than anything else, and the fix ended up having nothing to do with what I kept trying to fix.

Edges from the same source node were stacking into one thick line. I tried different edge types, smoothstep and step. I built a custom component that drew right-angle paths and tried to spread edges to different handle positions along node height. Some things helped in some cases, made others worse, and the code got worse with every change.

What I eventually figured out was that some of the arrows were wrong, not just overlapping. A connection like COMP 1805 → MATH 1104 → STAT 2605 was appearing because I was generating arrows by regex-parsing the raw prerequisites text on each course. The parser found course codes mentioned in a description and drew an arrow whether the implied relationship actually existed or not. The map looked like a rendering bug. It was a data bug.

The fix was to stop generating arrows from text. I added a program_edges table. Source course, target course, edge type, per program. A seeding script populated initial edges from the official Carleton maps, getting roughly 70% right, with the rest fixable via a PUT endpoint per program. Once set, stored permanently.

With correct explicit edges and single centered handles, React Flow's smoothstep routing worked fine without any help. The custom edge component got deleted. The whole thing got shorter and cleaner. I'd spent days trying to fix a rendering problem that didn't exist.

Deployment

Three platforms. Two failures.

Vercel first. The monorepo needed explicit config pointing to the right build directory. Got that wrong, fixed it, then hit Cloudflare proxy conflicts where the errors looked like application problems but weren't. Moved on after spending more time than I should have.

Render next, for the backend. I'd briefly tried Fly.io earlier and left a generated Dockerfile sitting in the repo. Render found it and used it instead of detecting a Python app. The deployment was broken in ways that didn't say why. Took a long time to find the file. Deleting it fixed everything immediately.

Railway just worked. Simple config, never had to think about it again.

What I learned

There's a pattern I kept running into. The arrow problem was in the data layer, not the rendering layer. The Render failure was in a forgotten file, not the application. The .env crash was a paste error, not a code bug. A lot of debugging time went to the right symptoms and the wrong cause.

Switching scrapers mid-project cost time upfront and saved more later. Keeping the repo clean isn't glamorous advice but a leftover Dockerfile genuinely cost me an afternoon. And at some point you ship something that isn't done. The site isn't perfect. It works, and 500+ students have found it useful.
